# AurumMQ PR4 — Internal Command Protocol Plan

Status: design plan  
Target area: `crates/aurum-internal-protocol`, `crates/aurum-types`, `crates/aurum-broker`, protocol adapters, and a minimal in-memory execution harness.  
Depends on: PR2 model testing and PR3 Rabbit-like delivery semantics in `aurum-core`.

---

## 1. Why PR4 exists

PR1–PR3 built the local core:

```text
HybridRangeBlockQueue
+ ConsumerSession
+ DeliveryWindow
+ Ack/Nack/Reject/Cancel semantics
```

That is still a local library API. The next architectural step is to define the stable internal language that protocol adapters, routing, runtime shards, and broker composition use to communicate.

PR4 creates that language:

```text
Protocol adapter
  -> Internal Command Protocol
  -> Routing / shard ownership
  -> aurum-core queue + consumer session
  -> Internal Event/Result batches
  -> Protocol adapter
```

This must happen before AMQP, native transport, storage, clustering, or thread-per-core runtime, because those modules should not call `aurum-core` directly or depend on each other's protocol-specific types.

The central goal:

> Build the adapter-neutral, batch-oriented command boundary of AurumMQ without contaminating `aurum-core` with AMQP/native/Kafka concepts.

## 3. Design thesis

The internal protocol is not a wire protocol. It is the in-process, cross-crate command model used by embedded adapters, sidecar gateways, routing, shard mailboxes, and future runtime backends.

It should be:

```text
batch-first
route-aware
shard-aware
protocol-neutral
payload-handle based
range/mask-first for settlement
static-dispatch friendly in hot paths
safe and ergonomic in cold/control paths
```

It should not be:

```text
AMQP-shaped
Kafka-shaped
HTTP-shaped
per-message enum lists
Vec<Box<dyn Command>>
string-heavy
payload-copying by default
storage-aware too early
cluster-consensus-aware too early
```

---

## 4. Architectural placement

Target data flow:

```text
AMQP adapter / Native adapter / future gateway
        |
        v
IngressCommandBatch
        |
        v
Compiled routing layer
        |
        v
ShardCommandBatch
        |
        v
Shard executor / thread-per-core runtime
        |
        v
aurum-core queue + consumer session
        |
        v
ShardEventBatch / CommandResultBatch
        |
        v
Protocol adapter response encoding
```

Important separation:

```text
IngressCommandBatch:
  commands before routing/shard ownership is fully resolved

ShardCommandBatch:
  commands already targeted to one shard/queue/session owner

ShardEventBatch:
  output generated by shard execution
```

This avoids forcing every protocol adapter to know the shard map and avoids forcing the core to know protocol details.

## 5. Dependency rules

### 5.1 Desired dependency graph

```text
aurum-types
  no project dependencies

auro-internal-protocol
  depends on aurum-types
  optionally smallvec/arrayvec/bitflags
  does NOT depend on aurum-core
  does NOT depend on AMQP/native protocol crates

aurum-core
  depends on aurum-types, aurum-intrusive, aurum-kernels
  does NOT depend on aurum-internal-protocol
  does NOT depend on AMQP/native protocol crates

aurum-protocol-native
  depends on aurum-internal-protocol
  does NOT depend directly on aurum-core

aurum-protocol-amqp
  depends on aurum-internal-protocol
  does NOT depend directly on aurum-core

aurum-broker / aurum-runtime
  compose aurum-internal-protocol + aurum-routing + aurum-core
```

### 5.2 Why `aurum-internal-protocol` must not depend on `aurum-core`

Adapters should be able to emit commands without importing the queue engine or consumer session implementation.

Bad:

```text
AMQP adapter -> aurum-core::ConsumerSession::ack(...)
```

Good:

```text
AMQP adapter -> AckCommandBatch -> broker/shard executor -> aurum-core
```

This lets us later run adapters as:

```text
embedded adapter
sidecar gateway
external gateway
```

without changing the core.


2. Keep PR3 request structs in `aurum-core` for now if moving them would be large.
3. Define command-protocol request structs in `aurum-internal-protocol` with equivalent semantics.
4. Add conversion helpers in `aurum-broker` or a small bridge module, not in adapters.

The design target is no cycles.

## 7. Core design constraints

### 7.1 Batch-first, not command-list-first

Do not model the hot path as:

```rust
Vec<ShardCommand>
```

where each item is a different enum variant.

Prefer:

```rust
pub enum ShardCommandBatch<P = PayloadHandle> {
    Publish(PublishBatch<P>),
    Credit(CreditBatch),
    Ack(AckCommandBatch),
    Nack(NackCommandBatch),
    Cancel(CancelConsumerBatch),
}
```

The enum dispatch happens once per batch, not once per message.

### 7.2 No payload copies in the internal command protocol

The protocol should pass payload handles/descriptors, not `Vec<u8>` per message.

Initial PR4 can define lightweight descriptors:

```rust
pub struct PayloadHandle(pub u64);

pub struct PayloadRef {
    pub handle: PayloadHandle,
    pub len: u32,
    pub class: PayloadClass,
}
```

Later backends can map this to:

```text
registered buffer lease
Bytes/BytesMut region
segment-log reference
inline tiny payload
external gateway buffer ID
```

### 7.3 Range/mask-first settlement

Settlement commands should align with `aurum-core`:

```text
AckRange
AckMask
NackMask
NackRange
RejectRange
RejectMask
```

Adapters may receive individual delivery tags, but before hitting the shard executor they should be coalesced whenever possible.

### 7.4 Protocol-neutral semantics

Internal commands should not contain names like:

```text
basic.ack
basic.nack
channel.close
KafkaFetch
KafkaProduce
```

They should contain semantic commands:

```text
Ack
Nack
Reject
Publish
ConsumeStart
CreditUpdate
CancelConsumer
ResolveRoute
```

Protocol adapters map external operations to these.

## 8. Static dispatch vs dynamic dispatch policy

### 8.1 Hot path policy

Hot path code should prefer:

```text
enums for closed command families
generics for backend specialization
const generics for fixed-capacity batches when useful
bitflags for boolean policy/state flags
concrete structs
SmallVec/ArrayVec for small common-case batches
```

Avoid:

```text
dyn Trait per message
dyn Trait per ack
Box<dyn Command>
HashMap lookups per message when route_id exists
heap allocation per message
string routing per message
```

### 8.2 Cold/control path policy

Dynamic dispatch is acceptable for:

```text
protocol plugin registry
external gateway adapters
admin API handlers
metrics exporters
configuration providers
future storage backends during initialization
```

Example acceptable cold path:

```rust
pub trait GatewayPlugin {
    fn name(&self) -> &'static str;
    fn start(&self, ctx: GatewayContext) -> Result<(), GatewayError>;
}
```

Not acceptable in hot path:

```rust
pub trait Command {
    fn execute(&self, shard: &mut Shard);
}

Vec<Box<dyn Command>>
```

### 8.3 Enum-based dispatch for batch families

A single batch enum is fine:

```rust
match batch {
    ShardCommandBatch::Publish(batch) => exec_publish(batch),
    ShardCommandBatch::Ack(batch) => exec_ack(batch),
    ShardCommandBatch::Nack(batch) => exec_nack(batch),
    ShardCommandBatch::Credit(batch) => exec_credit(batch),
    ShardCommandBatch::Cancel(batch) => exec_cancel(batch),
}
```

This branch is once per batch and likely predictable. It also keeps the API ergonomic.

## 9. Enums and bitflags design

The project prefers enums and bitflags when they model real states and policies.

### 9.1 Enums for closed state spaces

Good enum candidates:

```rust
pub enum PublishTarget {
    Route(RouteId),
    Queue(QueueId),
    ExchangeKey(ExchangeKeyRef),
}

pub enum SettlementMode {
    One,
    Multiple,
}

pub enum RequeuePolicy {
    Requeue,
    DeadLetter,
    Drop,
}

pub enum CommandBatchKind {
    Publish,
    Ack,
    Nack,
    Credit,
    Consume,
    Cancel,
    ResolveRoute,
}

pub enum PayloadClass {
    Empty,
    Tiny,
    Small,
    Medium,
    Large,
    External,
}
```

Use `#[repr(u8)]` for very hot/tiny enums if layout matters and there is a clear reason.

### 9.2 Bitflags for combinable options

Good bitflag candidates:

```rust
PublishFlags:
  PERSISTENT
  MANDATORY
  CONFIRM_REQUIRED
  ROUTED_BY_ID
  HAS_HEADERS
  HAS_EXPIRATION
  HAS_PRIORITY

MessageFlags:
  REDELIVERED
  COMPRESSED
  TRACED
  HAS_PAYLOAD_REF

AckFlags:
  MULTIPLE
  FROM_ADAPTER_COALESCER

NackFlags:
  MULTIPLE
  REQUEUE
  DEAD_LETTER

ConsumeFlags:
  EXCLUSIVE
  NO_LOCAL
  AUTO_ACK

CommandFlags:
  LOW_LATENCY
  BEST_EFFORT
  REQUIRES_CONFIRM
```

Policy:

```text
Enums express one-of-N.
Bitflags express many independent boolean features.
Do not replace meaningful state machines with unstructured flags.
```

## 10. Internal command taxonomy

PR4 should define two primary command layers.

### 10.1 Ingress commands

Ingress commands are emitted by protocol adapters before full routing/shard ownership resolution.

```rust
pub enum IngressCommandBatch<P = PayloadHandle> {
    Publish(IngressPublishBatch<P>),
    ResolveRoute(ResolveRouteBatch),
    Control(ControlCommandBatch),
}
```

Initial scope:

```text
Publish by route_id
Publish by queue_id for tests
Publish by exchange key placeholder
ResolveRoute placeholder
```

Out of scope for PR4:

```text
full declarations
full permissions
topology mutation
AMQP channel lifecycle
```

### 10.2 Shard commands

Shard commands are already targeted to one owner shard.

```rust
pub enum ShardCommandBatch<P = PayloadHandle> {
    Publish(ShardPublishBatch<P>),
    Consume(ConsumeCommandBatch),
    Credit(CreditCommandBatch),
    Ack(AckCommandBatch),
    Nack(NackCommandBatch),
    Reject(RejectCommandBatch),
    Cancel(CancelConsumerBatch),
}
```

The runtime mailbox should eventually carry `ShardCommandBatch`, not arbitrary protocol frames.

## 11. Publish command design

### 11.1 Publish target split

Before routing:

```rust
pub enum IngressPublishTarget {
    Route(RouteId),
    Queue(QueueId),
    ExchangeKey(ExchangeKeyRef),
}
```

After routing:

```rust
pub struct ShardPublishTarget {
    pub queue_id: QueueId,
    pub route_id: Option<RouteId>,
}
```

`RouteId` should be the native fast path. `ExchangeKeyRef` exists for AMQP-like paths and route resolution.

### 11.2 Publish batch shape

Initial ergonomic version:

```rust
pub struct PublishBatch<P = PayloadHandle> {
    pub batch_id: BatchId,
    pub source: SourceId,
    pub target: PublishTarget,
    pub flags: PublishFlags,
    pub records: SmallVec<[PublishRecord<P>; 32]>,
}

pub struct PublishRecord<P = PayloadHandle> {
    pub payload: P,
    pub payload_len: u32,
    pub message_flags: MessageFlags,
    pub priority: u8,
    pub expiration_ms: Option<u32>,
    pub key_hash: u32,
}
```

This is AoS. That is acceptable for PR4.

Future optimization:

```text
PublishBatchSoA
  payload_handles[]
  payload_lens[]
  flags[]
  priorities[]
  key_hashes[]
```

Do not prematurely implement SoA command batches before we have real adapter/runtime benchmarks. The queue engine remains DOD; command protocol can start ergonomic and evolve.

### 11.3 Confirm semantics placeholder

Publish commands need confirm intent:

```rust
pub enum ConfirmMode {
    None,
    Accepted,
    LocalDurable,
    Quorum,
}
```

For PR4, only define it. Actual durable/quorum semantics are later PRs.

## 13. Settlement command design

Settlement should be expressible at multiple compactness levels.

### 13.1 Ack

```rust
pub struct AckCommandBatch {
    pub consumer_id: ConsumerId,
    pub flags: AckBatchFlags,
    pub items: SmallVec<[AckCommand; 8]>,
}

pub enum AckCommand {
    Tag {
        tag: DeliveryTag,
        mode: SettlementMode,
    },
    Range {
        start: DeliveryTag,
        end: DeliveryTag,
    },
    Mask {
        base: DeliveryTag,
        mask: u64,
    },
}
```

Rationale:

```text
Tag:
  easiest bridge from AMQP basic.ack

Range:
  common after adapter coalescing

Mask:
  useful for sparse tag windows and native protocol
```

The shard executor converts tag-based forms into `AckRange`/`AckMask`/core operations through `ConsumerSession`.

### 13.2 Nack

```rust
pub struct NackCommandBatch {
    pub consumer_id: ConsumerId,
    pub flags: NackBatchFlags,
    pub items: SmallVec<[NackCommand; 8]>,
}

pub enum NackCommand {
    Tag {
        tag: DeliveryTag,
        mode: SettlementMode,
        disposition: NackDisposition,
    },
    Range {
        start: DeliveryTag,
        end: DeliveryTag,
        disposition: NackDisposition,
    },
    Mask {
        base: DeliveryTag,
        mask: u64,
        disposition: NackDisposition,
    },
}

pub enum NackDisposition {
    Requeue,
    DeadLetter,
    Drop,
}
```

### 13.3 Reject

Reject is semantically a single-message settlement in Rabbit-like terms, but internally we can still model it as a settlement command.

```rust
pub struct RejectCommandBatch {
    pub consumer_id: ConsumerId,
    pub items: SmallVec<[RejectCommand; 8]>,
}

pub struct RejectCommand {
    pub tag: DeliveryTag,
    pub disposition: NackDisposition,
}
```

`Reject` can map to `NackCommand::Tag { mode: One, ... }` internally if we want fewer code paths.

## 14. Event/result design

Commands need outputs. PR4 should define event batches, even if execution is minimal.

### 14.1 Shard events

```rust
pub enum ShardEventBatch<P = PayloadHandle> {
    PublishConfirm(PublishConfirmBatch),
    Delivery(DeliveryEventBatch<P>),
    Settlement(SettlementResultBatch),
    Consumer(ConsumerEventBatch),
    Error(CommandErrorBatch),
}
```

### 14.2 Delivery events

Delivery events should preserve segment/range/mask nature.

```rust
pub struct DeliveryEventBatch<P = PayloadHandle> {
    pub consumer_id: ConsumerId,
    pub first_tag: DeliveryTag,
    pub segments: SmallVec<[DeliveryEventSegment<P>; 8]>,
}

pub enum DeliveryEventSegment<P = PayloadHandle> {
    Range {
        start_tag: DeliveryTag,
        start_seq: u64,
        len: u32,
        payloads: PayloadSlice<P>,
        flags: DeliveryEventFlags,
    },
    Mask {
        base_tag: DeliveryTag,
        block: u32,
        word: u8,
        mask: u64,
        payloads: PayloadSlice<P>,
        flags: DeliveryEventFlags,
    },
}
```

For PR4, payload slices can be placeholder handles.

### 14.3 Publish confirms

```rust
pub struct PublishConfirmBatch {
    pub source: SourceId,
    pub batch_id: BatchId,
    pub accepted: u32,
    pub first_seq: Option<u64>,
    pub mode: ConfirmMode,
}
```

### 14.4 Errors

Errors must be structured:

```rust
pub enum CommandErrorKind {
    InvalidRoute,
    StaleRouteEpoch,
    QueueNotFound,
    ConsumerNotFound,
    InvalidDeliveryTag,
    CreditExceeded,
    PermissionDenied,
    Backpressure,
    InternalInvariantViolation,
}
```

Error should contain enough protocol-neutral information for adapters to map to AMQP channel error, native error frame, or gateway error.

## 15. ID and epoch design

PR4 should prepare for routing and failover without implementing them fully.

Add or reserve types:

```rust
pub struct BatchId(u64);
pub struct CommandId(u64);
pub struct SourceId(u64);
pub struct ConnectionId(u64);
pub struct RouteEpoch(u32);
pub struct ShardEpoch(u32);
```

Route-aware commands should include optional epoch:

```rust
pub struct RouteRef {
    pub route_id: RouteId,
    pub epoch: RouteEpoch,
}
```

Shard-targeted commands can later include:

```rust
pub struct ShardRef {
    pub shard_id: ShardId,
    pub epoch: ShardEpoch,
}
```

PR4 does not implement stale epoch handling, but the data model must not make it impossible.

## 16. Payload handle design

Define a neutral handle:

```rust
#[repr(transparent)]
pub struct PayloadHandle(pub u64);

pub struct PayloadDescriptor {
    pub handle: PayloadHandle,
    pub len: u32,
    pub class: PayloadClass,
    pub flags: PayloadFlags,
}
```

Possible future handle encodings:

```text
RX buffer lease
registered io_uring buffer id
payload arena offset
segment log ref
gateway sidecar buffer id
inline small payload table id
```

Do not put `Vec<u8>` in `PublishRecord` hot structs.

For tests/examples, a simple mock payload registry can map `PayloadHandle -> Vec<u8>` outside hot structs.

## 17. Execution bridge design

PR4 should include a minimal bridge, likely in `aurum-broker` or `aurum-runtime`, not in `aurum-internal-protocol`.

Suggested module:

```text
crates/aurum-broker/src/shard_executor.rs
```

Prototype:

```rust
pub struct InMemoryShardExecutor {
    queue: HybridRangeBlockQueue,
    consumers: ConsumerTable,
}

impl InMemoryShardExecutor {
    pub fn execute<P>(&mut self, batch: ShardCommandBatch<P>, events: &mut Vec<ShardEventBatch<P>>);
}
```

Initial scope:

```text
Publish -> queue.publish_contiguous(records.len())
ConsumeStart -> create ConsumerSession
Credit -> update session credit
Ack -> apply through ConsumerSession
Nack -> apply through ConsumerSession
Cancel -> session cancel and core requeue/drop placeholder
```

This bridge is not the final runtime. It is the first proof that:

```text
adapter-neutral commands can drive the core
```

## 18. Backend strategy

### 18.1 Command sink backends

Protocol adapters need somewhere to send command batches. We can use a trait, but not per message.

Cold/edge acceptable:

```rust
pub trait CommandSink<P> {
    fn submit_ingress(&mut self, batch: IngressCommandBatch<P>) -> Result<(), SubmitError>;
}
```

For embedded native/AMQP fast paths, prefer static dispatch:

```rust
pub struct NativeAdapter<S> {
    sink: S,
}

impl<S: CommandSink<PayloadHandle>> NativeAdapter<S> { ... }
```

For plugins/gateways:

```rust
Box<dyn GatewayPlugin>
```

is acceptable because this is lifecycle/cold registration, not per-message execution.

### 18.2 Executor backends

Eventually there may be:

```text
InMemoryShardExecutor
StorageBackedShardExecutor
ReplicatedShardExecutor
```

Do not use `dyn ShardExecutor` inside the hot shard loop by default.

Prefer:

```rust
ShardRuntime<E: ShardExecutorBackend>
```

or an enum if the set is closed:

```rust
pub enum ExecutorBackend {
    InMemory(InMemoryShardExecutor),
    StorageBacked(StorageBackedShardExecutor),
}
```

Given your preference, use enums when the set is closed and meaningful; use generic static dispatch when benchmarking shows inlining matters; use dynamic dispatch only at setup/registry boundaries.

## 21. Performance plan

PR4 is mostly boundary design, but should avoid known bad patterns.

### 21.1 Allocation policy

Common-case command batches should use:

```text
SmallVec<[T; 8]>
SmallVec<[T; 16]>
SmallVec<[T; 32]>
ArrayVec where strict fixed capacity is desired
```

Avoid:

```text
Vec allocation per message
String per routing key in hot path
HashMap per command batch
Box<dyn Command>
```

### 21.2 Benchmark target

H3 should measure:

```text
command construction ns/message
execute publish command ns/message
execute ack command ns/message
execute nack/redelivery command ns/message
extra overhead vs direct aurum-core calls
```

Acceptance target:

```text
Internal command protocol should add minimal overhead per batch.
If it adds meaningful overhead per message, redesign before AMQP/native.
```

### 21.3 Profiling concern

If `SmallVec` overhead appears in hot loops, introduce specialized batch structs for publish/ack paths.

Do not optimize with unsafe until:

```text
correctness tests pass
perf shows clear bottleneck
assembly confirms bounds/check overhead
```

## 22. Backpressure and scalability hooks

PR4 should not implement full backpressure, but the command model must leave room for it.

Include placeholders:

```text
CommandResult::Accepted
CommandResult::RejectedBackpressure
CommandResult::StaleEpoch
CommandResult::Redirect
```

Future runtime needs:

```text
mailbox full
shard overloaded
storage lagging
replica lagging
consumer socket blocked
```

So command submission should be fallible:

```rust
pub enum SubmitError {
    Backpressure,
    Closed,
    StaleEpoch,
    ShardUnavailable,
}
```

Do not make `submit()` infallible in public traits.

## 23. Route/shard epoch hooks

Even though PR4 does not implement failover, command structs should allow stale detection later.

Add optional fields:

```rust
pub struct RoutedTarget {
    pub route_id: RouteId,
    pub route_epoch: RouteEpoch,

pub struct ShardTarget {
    pub shard_id: ShardId,
```

Do not force all tests to use real epochs yet; use `RouteEpoch::ZERO` / `ShardEpoch::ZERO` or default values.

## 28. Design mantra for PR4

```text
Adapters speak protocol.
Routing speaks routes.
Runtime speaks shards.
Core speaks ranges and masks.
Internal protocol connects them without leaking one layer into another.
```

If a type smells like AMQP, it belongs in `aurum-protocol-amqp`.
If a type smells like TCP/QUIC bytes, it belongs in transport/protocol crates.
If a type smells like queue state, it belongs in `aurum-core`.
If a type is the semantic command connecting layers, it belongs in `aurum-internal-protocol`.
